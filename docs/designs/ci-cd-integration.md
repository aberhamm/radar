## Vision

### 10x Check
The 10x version: radar runs in every client's CI pipeline automatically. Every PR gets a scorecard. Findings trend down over time. Consultants don't manually run audits — they configure the pipeline once and monitor. The tool becomes a continuous quality signal, not a point-in-time snapshot.

### Approach Selected
**Approach C: GitHub + Azure DevOps + Trend Tracking + Docker** — Covers both active client platforms. Findings diff shows New/Resolved/Persistent across runs. Docker image on GHCR for fast, reliable CI execution. Quality gates are configurable.

## Architecture

### Adapter Pattern

```
src/ci/adapter.ts (CiPlatformAdapter interface)
│
├── detect(): Reads env vars, returns appropriate adapter
│   GITHUB_ACTIONS=true   → GitHubCiAdapter
│   TF_BUILD=True         → AzureDevOpsCiAdapter
│   GITLAB_CI=true        → future GitLab adapter
│   none                  → GenericAdapter (exit code + stdout only)
│
├── interface methods:
│   postComment(body: string): Promise<string | null>
│   updateComment(marker: string, body: string): Promise<string | null>
│   postAnnotations(findings: Finding[], cap?: number): Promise<number>
│   addLabels(labels: string[]): Promise<void>
│   uploadSarif(sarif: SarifLog): Promise<void>
│   setStatus(state: 'success'|'failure'|'pending', desc: string): Promise<void>
│   downloadPreviousArtifact(name: string): Promise<string | null>
│   uploadArtifact(name: string, path: string): Promise<void>
│   getCapabilities(): CiCapabilities
│
└── CiCapabilities: { canComment, canAnnotate, canLabel, canUploadSarif, canSetStatus, canManageArtifacts }
```

### GitHub Adapter
- Auth: `GITHUB_TOKEN` env var (provided by Actions)
- API: GitHub REST API v3 via native `fetch()`
- PR context: parsed from `$GITHUB_EVENT_PATH` (pull_request event payload)
- Annotations: Checks API (Create Check Run with annotations)
- SARIF: `POST /repos/{owner}/{repo}/code-scanning/sarifs`
- Artifacts: GitHub REST API via native `fetch()` (consistent with all other API calls, no `@actions/artifact` dependency)
- Comment marker: `<!-- radar-ci-comment -->` for update-in-place
- SARIF: `canUploadSarif` defaults to `false`. On first upload attempt, if 403 (no GitHub Advanced Security), log warning and set `canUploadSarif=false` for the rest of the run. Many client repos on free/Team plans will not have Advanced Security enabled.

### Azure DevOps Adapter
- Auth: `$(System.AccessToken)` via `SYSTEM_ACCESSTOKEN` env var
- API: Azure DevOps REST API v7.1 via native `fetch()`
- PR context: `SYSTEM_PULLREQUEST_PULLREQUESTID` env var
- Org/project: `SYSTEM_COLLECTIONURI` + `SYSTEM_TEAMPROJECT` env vars
- Annotations: PR thread comments anchored to file+line
- SARIF: Not supported (skip silently)
- Artifacts: Pipeline Artifacts API
- Comment marker: same HTML comment for update-in-place
- Capabilities probe at init: one API call to discover permissions. Optional features (labels, status) skip silently if 403.
- Comment search pagination: `$top=200&$skip=N` to find `<!-- radar-ci-comment -->` marker. Cap at 1000 threads to avoid runaway pagination on noisy PRs.

### Data Flow

```
radar analyze --json
     │
     ▼
  Run agent (existing)
     │
     ▼
  adapter = detectCiPlatform()
     │
     ├── adapter.downloadPreviousArtifact('radar-findings')
     │        → returns previous findings.json or null
     │
     ├── if previous: diff(current, previous) → trend summary
     │        → New/Resolved/Persistent via fingerprint matching
     │
     ├── render PR comment (scorecard + trend + collapsible findings)
     │        → progressive truncation if >60K chars
     │
     ├── adapter.updateComment('<!-- radar-ci-comment -->', body)
     │        → if no existing comment, falls back to postComment
     │
     ├── adapter.postAnnotations(findings, cap=30)
     │        → sorted by severity, capped
     │
     ├── adapter.addLabels(deriveLabels(findings))
     │        → skipped if capabilities probe says no permission
     │
     ├── adapter.uploadSarif(generateSarif(findings))
     │        → GitHub only, skip on Azure DevOps
     │
     ├── adapter.uploadArtifact('radar-findings', currentFindings.json)
     │        → for next run's trend tracking
     │
     ├── webhook.send(summary) if WEBHOOK_URL set
     │
     └── qualityGate.evaluate(scorecard, config) → exit code
```

## Security Requirements

1. **Token masking** — `maskToken()` in `src/ci/utils.ts` replaces all but last 4 chars with `****`. Used in all error messages.
2. **Never log tokens** — GITHUB_TOKEN, SYSTEM_ACCESSTOKEN never appear in stdout, stderr, JSON output, SARIF, PR comments, or webhook payloads.
3. **Webhook URL validation** — validate against existing `domainBlocklist.ts` before sending.
4. **No secrets in artifacts** — uploaded findings.json contains finding data only, no tokens or env vars.

## Quality Gate Config

**Exit code semantics:** Quality gate evaluation produces the exit code. `failOn` match returns exit 1, `warnOn` match returns exit 0 with a warning in the PR comment. This replaces the current hardcoded `red=1` logic in `analyze.ts:247-250`. Exit code 2 remains reserved for agent errors (non-gate failures).
