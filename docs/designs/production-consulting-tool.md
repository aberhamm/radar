## Vision

### 10x Check
The 10x version isn't "better reports." It's a self-service platform where any consultant creates a goal type, runs it against any repo, and gets a consulting deliverable they can hand to a client. The tool becomes infrastructure that generates billable hours. One agent architecture, many consulting deliverables. Same tools, different rules, different output.

### Approach Selected
**Approach C: Gauntlet + Docs** — Prove quality on real client repos first (Phase 1), then build team-ready distribution (Phase 2). Quality before distribution. If teammates run this on a client repo and it produces bad findings, trust is lost permanently.

### Gauntlet Pass Criteria (Quantitative Bar)
- Zero crashes on any repo
- Zero hallucinated evidence (line numbers must exist in the actual file — enforced by existing evidence verification system in `src/tools/analysis/recordFinding.ts` which checks line existence at recording time)
- <20% of findings with confidence <=3
- All brief sections populated
- Scorecard has no "unknown" categories

### Phase 1 Exit Gate
Phase 1 is complete when all 5+ repos pass all gauntlet criteria with zero open regressions. The gauntlet tracker (`radar gauntlet` CLI command) must show all-green before Phase 2 begins.

## Scope Boundaries
- **Findings diff** compares two runs of the **same** repo (temporal tracking). **Benchmarking** compares across different repos (cross-repo ranking). The diff feature does NOT extend to cross-repo comparison — that's the deferred benchmarking feature.
- **Goal Builder generated files** must pass the gauntlet before being added to the template library. A generated goal-*.md is a draft until validated by a real agent run.

## Phase Sequencing
Phase 1 must complete before Phase 2 begins. Phase 1 exit gate (see above) is the concrete trigger. Phase 1 findings will inform Phase 2 scope (e.g., if certain goal types consistently fail, the template library should exclude them).

## Artifact Specs

### Gauntlet Tracker
- **Storage:** `output/gauntlet-results.jsonl` — one line per run
- **Schema:** `{ repo: string, goal: string, timestamp: string, crashFree: boolean, hallucinationFree: boolean, confidencePass: boolean, sectionsPopulated: boolean, scorecardComplete: boolean, findingsCount: number, notes?: string }`
- **CLI:** `radar gauntlet` prints a table:

```
  Repo                    | Goal       | Crash | Halluc | Conf | Sections | Score | Status
  ------------------------|------------|-------|--------|------|----------|-------|-------
  client-a (sitecore)     | onboarding | PASS  | PASS   | PASS | PASS     | PASS  | GREEN
  client-a (sitecore)     | audit      | PASS  | PASS   | FAIL | PASS     | PASS  | YELLOW
  client-b (optimizely)   | onboarding | PASS  | PASS   | PASS | PASS     | PASS  | GREEN
```

### Findings Diff
- **Command:** `radar diff <run-a-dir> <run-b-dir>`
- **Matching key:** Finding fingerprint (SHA-256 of `category:firstFilePath:normalizedTitle`)
- **Output:** Markdown table to stdout + optional JSON with `--json` flag
- **JSON schema:** `{ repo: string, runA: { dir: string, timestamp: string }, runB: { dir: string, timestamp: string }, new: Finding[], resolved: Finding[], persistent: Finding[] }`

```
  ## Findings Diff: client-a (2026-04-01 -> 2026-04-07)

  ### New (3)
  | Finding | Severity | Category |
  |---------|----------|----------|
  | STALE-SDK-VERSION | high | dependencies |

  ### Resolved (1)
  | Finding | Severity | Category |
  |---------|----------|----------|
  | MISSING-ENV-VAR | medium | configuration |

  ### Persistent (8)
  (unchanged findings, listed for completeness)
```

### Webhook Notification
- **Trigger:** Post-run, if `WEBHOOK_URL` env var is set (consistent with existing `.env` pattern)
- **Payload (summary-only, no sensitive data):**

```json
{
  "tool": "radar",
  "repo": "client-a",
  "goal": "onboarding",
  "score": "B-",
  "findingsCount": 14,
  "criticalCount": 2,
  "topRisks": ["Outdated SDK", "Missing preview config", "No error boundary"],
  "duration": "8m 23s",
  "cost": "$0.74",
  "timestamp": "2026-04-07T12:00:00Z"
}
```
- **Behavior:** Fire-and-forget POST with 5s timeout. Failure logs a warning, never blocks the run.
- **Security:** Webhook URL is a secret — read from env var, never committed. Teams sharing config should use per-machine env vars, not shared .env files. URL is validated against domainBlocklist.ts (reuse existing SSRF protection).

### Client-Ready PDF Export
- **Trigger:** `--pdf` flag on `radar analyze`
- **Library:** `@react-pdf/renderer` (pure Node, no Chrome binary, cross-platform). Chosen over Puppeteer to avoid 300MB Chromium download and CI compatibility issues.
- **Output:** `output/{timestamp}/report.pdf`
- **Sections included:** Executive summary, scorecard visualization, all findings (grouped by category with severity badges), documentation references, recommended next steps
- **Sections excluded:** Investigation log, raw tool call traces, agent reasoning, per-step evidence (internal-only detail)
- **Branding:** Configurable via `config/pdf-branding.json`. Fields: `{ logoPath?: string, companyName?: string, accentColor?: string, footerText?: string }`. All optional — defaults to plain/generic if file is missing or fields are absent.
- **Optional dependency:** `@react-pdf/renderer` is listed in `peerDependencies` (not required). `--pdf` flag checks for installation and prints `"PDF export requires @react-pdf/renderer. Run: pnpm add @react-pdf/renderer"` if missing.

### Goal Builder
- **Command:** `radar goal-builder`
- **Questions:** Stored in `config/goal-builder-questions.json` (editable config file, no code change required to update questions). Not versioned/pinned — the config file IS the current version.
- **Output:** Writes to `src/rules/goal-{name}.md` after user confirms
- **Validation:** Generated markdown is passed through the existing rule loader to verify structure before writing. Invalid output shows errors and re-prompts (max 2 retries). After 2 failed attempts, writes the draft to `src/rules/goal-{name}.draft.md` and exits with a non-zero code.

### Template Library
- **Config:** `config/templates.json` — array of preset objects
- **Schema:** `{ name: string, description: string, goal: GoalType, budget: number, platform?: string }`
- **CLI:** `--template <name>` loads the preset. Explicit CLI flags override template defaults.
- **Built-in presets:** Quick Security Scan (15 calls), Deep Audit (60 calls), New Dev Onboarding (45 calls), Migration Readiness (50 calls)
- **Custom templates:** Teams add entries to `config/templates.json`. Validated at load time with Zod against the schema above. If a template is invalid, `radar analyze` exits with a descriptive error.
- **Implementation note (PDF):** Use dynamic import for optional dependency: `const mod = await import('@react-pdf/renderer').catch(() => null)`. Do not use static import at module top-level (breaks ESM if package is not installed).

### Issues to address before implementation

1. **Zod validation for gauntlet JSONL reads** — corrupt lines should warn, not crash. (10 LOC)
2. **Handle null fingerprints in diff command** — old runs may lack fingerprints. Skip with warning or hash on the fly. (5 LOC)
3. **PDF conditional test pattern** — `describe.skipIf(!hasReactPdf)` for CI environments where peer dep isn't installed. (1 line)
4. **Validate GoalType enum in template Zod schema** — catch invalid goal references at load time, not runtime. (3 LOC)
5. **Integration point** — gauntlet logger and webhook fire from CLI layer (`src/index.ts` after `runAgent()` returns), not from runner.ts. Runner stays pure.
6. **Create `config/` directory** — plan references 3 config files (`templates.json`, `goal-builder-questions.json`, `pdf-branding.json`) but directory doesn't exist yet.

### Implementation order

1. Gauntlet tracker (standalone, no dependencies)
2. Findings diff (standalone, uses existing fingerprints)
3. Run gauntlet against first real repo

**Phase 2:**
