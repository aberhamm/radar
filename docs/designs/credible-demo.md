## Vision

### 10x Check
The 10x version isn't "better audit output." It's "an agent runtime platform that plugs into any consulting workflow." The demo proves the concept with repo audits, but the architecture (deterministic tools + markdown rules + agentic orchestration + pluggable providers) is a template for any consulting deliverable: security reviews, migration plans, dependency audits, compliance checks. Each one is a new goal type + rule file, zero tool changes.

4. GitHub issue/PR comment output (visible, shareable)

## Error Recovery Design
- Agent loop errors: catch, log to output/debug-{timestamp}.json, auto-assemble partial brief
- LLM malformed response: retry 1x, then treat as agent error
- LLM refusal: log, skip this tool call, continue loop
- LLM empty response: retry 1x, then treat as agent error
- GitHub hook failures: warn, skip posting, still write files locally
- Comparison engine: if one repo fails, produce single report + error note

## Run Metrics Schema

Computed post-run from AgentState counters + Portkey response usage headers. Not a replacement for AgentState tracking... it's a summary layer on top.

```typescript
interface RunMetrics {
  startedAt: string;
  completedAt: string;
  durationMs: number;
  toolCalls: number;  // sum from AgentState.toolCallCount
  models: {
    // Key = short alias: "sonnet-4.6" or "haiku-4.5" (mapped from Bedrock model IDs)
    [modelAlias: string]: {
      bedrockModelId: string;  // e.g. "us.anthropic.claude-sonnet-4-6-v1:0"
      calls: number;
      inputTokens: number;
      outputTokens: number;
      cachedTokens: number;
      estimatedCostUsd: number;  // from config/model-pricing.json, see risks
    };
  };
  totalEstimatedCostUsd: number;
}
```

## Design Details (per accepted scope item)

### 1. Interactive Demo Mode
- **CLI flag:** `--verbose` (chosen over `--interactive` to avoid implying user interaction during the run)
- **Implementation:** Event emitter on the agent runner. Each tool call emits `{ step, toolName, reasoning, result, error?: string }`. In verbose mode, a console renderer formats and prints each event to stdout as it happens. Failed tool calls show the error field in red. In non-verbose mode, events are silently collected into the investigation log.
- **Pi dependency:** If Pi supports tool call callbacks/hooks, use those. If Pi only returns final results, use the DirectLoopRunner (the manual tool-calling loop from the design doc's Chunk 0 fallback, where each loop iteration is naturally visible). Validate Pi callback support in Chunk 0 spike.
- **New files:** `src/output/interactiveRenderer.ts`

### 2. Multi-Provider Interface
- **What it is:** Extend the existing `ModelProvider` interface from spec section 14. Add a `createProvider(config)` factory function that reads provider type from config/env.
- **Portkey:** Fully implemented, primary for demo.
- **Stubs:** `src/providers/stub.ts` exports a single `StubProvider` class (not per-provider stubs) that throws `"Provider not implemented: ${name}. See docs for setup."` Any provider name other than "portkey" routes here.
- **No YAGNI:** We're not building Copilot/Codex adapters. We're building the interface that makes adding them trivial. One stub file, ~10 lines.
- **New files:** `src/providers/factory.ts`, `src/providers/stub.ts`

### 3. CI/CD Quality Gate
- **GoalType extension:** Add `'ci-check'` to the `GoalType` union in `src/types/state.ts`
- **Rule file:** `src/rules/goal-ci-check.md` with lighter investigation scope (skip preview/editing, skip client questions, focus on security + dependencies + architecture)
- **Tool budget:** 25 calls (half of onboarding)
- **Output format:** Scorecard JSON + 1-paragraph summary string, both fields in a single JSON object written to stdout (for CI pipe usage) AND to `output/{timestamp}/ci-check.json`. No full narrative brief.
- **Exit code:** 0 = all green/yellow, 1 = any red category, 2 = agent error (partial output written, treat as "unknown" in CI)
- **Difference from audit:** CI-check is fast and binary (pass/fail). Audit is thorough and narrative.
- **New files:** `src/rules/goal-ci-check.md`, `src/output/compactRenderer.ts`

### 4. GitHub Output Hook
- **Trigger:** `--github-output` flag (opt-in, never automatic)
- **Auth:** Uses `gh` CLI (user's existing GitHub auth). No PAT token management.
- **Pre-flight:** Run `gh auth status` before attempting. If not authed, warn and skip.
- **For onboarding:** `gh issue create --title "Onboarding Brief: {repo}" --body "{brief}"`
- **For ci-check:** `gh pr comment {pr-number} --body "{scorecard + summary}"`. PR number from `--pr` flag or `GITHUB_PR_NUMBER` env var (standard in GitHub Actions as `${{ github.event.pull_request.number }}`).
- **Pre-flight:** Run `gh auth status` AND `gh repo view --json name` to verify both auth and repo write access. If either fails, warn and skip.
- **Error handling:** If gh fails for any reason, warn to stderr and continue. Output files always written locally regardless.
- **New files:** `src/output/githubHook.ts`

### 5. Comparison Report
- **What it compares:** Two repos against each other. Run the agent once per repo (sequentially), collect both outputs, then generate a comparison.
- **Trigger:** `npx tsx src/index.ts --compare <repo1> <repo2>`
- **Output location:** `output/{timestamp}/comparison.md` (alongside the individual `repo-a/` and `repo-b/` output directories)
- **Output schema:** Two scorecards side by side + delta table (which categories differ) + "key differences" section + "common patterns" section. Differences and common patterns are template-driven (not LLM-generated): the comparison renderer diffs the two scorecards and finding categories deterministically.
- **Partial failure:** If one repo fails, produce the successful report + "Comparison unavailable: {repo} failed: {reason}"
- **Demo strategy:** For live demo, pre-generate the comparison and show it. For async demo, run it fresh (6-10 min acceptable when not presenting live).
- **New files:** `src/output/comparisonRenderer.ts`

### 6. Budget Extension
- **Which budget:** Tool call budget only. Web search budget is irrelevant (stubbed). Token budget is managed by the LLM provider, not our code.
- **Interactive mode:** When `toolCallCount >= toolCallBudget`, prompt: "Budget reached ({n} calls). {m} findings so far. Continue for 50 more? [y/N]"
- **Non-interactive / CI mode:** Auto-assemble with current findings. No prompt, no hang.
- **Extension amount:** Fixed at 50 additional calls. Can extend multiple times.

## Assumptions and Risks

| Assumption | Risk if Wrong | Mitigation |
|-----------|---------------|------------|
| Pi supports tool call callbacks for interactive mode | Interactive mode requires DirectLoopRunner only | Validate in Chunk 0 spike. DirectLoopRunner always supports this natively. |
| Bedrock pricing is stable enough for cost estimates | ROI number is wrong in demo | Pricing stored in `config/model-pricing.json` (easy to update). Show token counts always, cost estimate with disclaimer: "estimated as of {date}." If pricing file missing, show tokens only. |
| `gh` CLI is installed on demo machine | GitHub hook silently fails | Pre-flight check. Local file output always works. |
| web_search is stubbed (no search API key) | CI/CD and migration goals produce thinner output | Acceptable for demo. Note in output: "web search disabled." |
| Comparison engine doubles LLM cost and time | Demo takes 6-10 min instead of 3-5 | Run comparison async or pre-generate for the demo. |
